SECTION 1: PROJECT OVERVIEW
================================================================================

Parseltongue is a code analysis toolkit that parses codebases into a graph
database (CozoDB) for efficient LLM-optimized querying.

Core value: 99% token reduction (2-5K tokens vs 500K raw dumps), 31x faster
than grep.

Version: 1.6.1 (HTTP-only architecture with always-on file watching + 7 graph
analysis algorithms)

Languages Supported: Rust, Python, JavaScript, TypeScript, Go, Java, C, C++,
Ruby, PHP, C#, Swift

Workspace Architecture:
  crates/
  ├── parseltongue/                        # CLI binary - dispatches to tools
  ├── parseltongue-core/                   # Shared types, traits, storage, tree-sitter parsing
  ├── pt01-folder-to-cozodb-streamer/      # Tool 1: Ingest codebase -> CozoDB
  └── pt08-http-code-query-server/         # Tool 8: HTTP REST API server

Dependency Flow: parseltongue (binary) -> pt01/pt08 (tools) -> parseltongue-core (shared)

Database: CozoDB with RocksDB backend
Naming Convention: FOUR-WORD naming for all functions/crates/commands
  Pattern: verb_constraint_target_qualifier()
  Example: filter_implementation_entities_only()

HTTP Server: 22 endpoints total on port 7777 (default)

9a. Academic Foundations
------------------------
- Dorothy E. Denning, "A Lattice Model of Secure Information Flow" (1976)
  - Communications of the ACM, 19(5):236-243
  - 2,079 citations, 137 highly influential
  - Foundational: security classes form a lattice, flows are partial orderings
  - Maps directly to Datalog relations in CozoDB

- Denning & Denning, "Certification of Programs for Secure Information Flow" (1977)
  - Static certification mechanism for flow policy compliance

- Andrew C. Myers, "JFlow: Practical Mostly-Static Information Flow Control" (POPL 1999)
  - Extended Java with statically-checked info flow annotations
  - Evolved into Jif (Java + Information Flow) at Cornell

- Sabelfeld & Myers, "Language-Based Information-Flow Security" (IEEE S&P 2003)
  - Canonical survey, ~150 papers cited

- Yamaguchi et al., "Modeling and Discovering Vulnerabilities with Code Property Graphs"
  (IEEE S&P 2014)
  - CPG = AST + CFG + PDG merged into single graph
  - IEEE Test-of-Time Award 2024
  - Taint pattern inference reduced code to inspect by 94.9%
  - Discovered 8 previously unknown vulnerabilities

9d. Datalog for Taint Analysis (Proven Superiority)
-----------------------------------------------------
- Souffle: Datalog engine for static program analysis
  - Compiles Datalog to parallel C++ via Futamura projection
  - Points-to analysis: 8 hours -> 20 minutes

SECTION 10: PROPOSED TAINT ANALYSIS ARCHITECTURE (FROM THESIS)
================================================================================

Three-layer architecture:
  Layer 1: tree-sitter extracts data-flow edges (AST -> DFG)
  Layer 2: Curated registry classifies sources/sinks/sanitizers
  Layer 3: CozoDB Datalog does propagation via graph reachability

No Z3 dependency. Curated sanitizer registry instead of symbolic reasoning.
Trade-off: ~15% more false positives for 12-language coverage + millisecond queries.

4 New CozoDB Relations:
  :create TaintSources { entity_key: String, source_kind: String, file: String, line: Int }
  :create TaintSinks { entity_key: String, sink_kind: String, cwe: String, file: String, line: Int, severity: String }
  :create DataFlowEdges { from_key: String, to_key: String, flow_type: String, file: String, from_line: Int, to_line: Int }
  :create Sanitizers { entity_key: String, sanitizer_name: String, protects_against: [String] }

Key Datalog Query (taint propagation):
  reachable[a, b, 1] := *DataFlowEdges[a, b, _, _, _, _]
  reachable[a, c, d] := reachable[a, b, d1], *DataFlowEdges[b, c, _, _, _, _], d = d1 + 1, d <= 20

  sanitized_path[source, sink, sink_kind] :=
      reachable[source, sanitizer_key, _],
      reachable[sanitizer_key, sink, _],
      *Sanitizers[sanitizer_key, _, protects],
      sink_kind in protects

  taint_flow[source_key, sink_key, source_kind, sink_kind, cwe, depth] :=
      *TaintSources[source_key, source_kind, _, _],
      *TaintSinks[sink_key, sink_kind, cwe, _, _, _],
      reachable[source_key, sink_key, depth],
      not sanitized_path[source_key, sink_key, sink_kind]

  /taint-source-sink-discovery
