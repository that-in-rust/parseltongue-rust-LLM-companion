## Pattern 1: XTask Pattern for Project Tooling
**File:** /Users/amuldotexe/Desktop/Pre20260126/OSS202601/Notes2026/rust-analyzer-patterns/xtask/src/main.rs
**Category:** Build Tooling

**Code Example:**
```rust
//! See <https://github.com/matklad/cargo-xtask/>.
//!
//! This binary defines various auxiliary build commands, which are not
//! expressible with just `cargo`. Notably, it provides tests via `cargo test -p xtask`
//! for code generation and `cargo xtask install` for installation of
//! rust-analyzer server and client.
//!
//! This binary is integrated into the `cargo` command line by using an alias in
//! `.cargo/config`.

mod codegen;
mod dist;
mod install;
mod release;
mod tidy;

fn main() -> anyhow::Result<()> {
    let flags = flags::Xtask::from_env_or_exit();

    let sh = &Shell::new()?;
    sh.change_dir(project_root());

    match flags.subcommand {
        flags::XtaskCmd::Install(cmd) => cmd.run(sh),
        flags::XtaskCmd::Release(cmd) => cmd.run(sh),
        flags::XtaskCmd::Dist(cmd) => cmd.run(sh),
        flags::XtaskCmd::Codegen(cmd) => cmd.run(sh),
        flags::XtaskCmd::Tidy(cmd) => cmd.run(sh),
    }
}

fn project_root() -> PathBuf {
    let dir =
        env::var("CARGO_MANIFEST_DIR").unwrap_or_else(|_| env!("CARGO_MANIFEST_DIR").to_owned());
    PathBuf::from(dir).parent().unwrap().to_owned()
}
```

**Why This Matters for Contributors:** The xtask pattern is the standard Rust approach for custom build tooling that goes beyond what `cargo` supports natively. It's invoked via `.cargo/config` aliases, providing a discoverable, type-safe interface for build tasks. This pattern replaces shell scripts with Rust code that can be tested and refactored.

**Rust-Specific Insight:**
The xtask pattern exemplifies Rust's philosophy of "tools written in the language they serve." Key idioms:
- **RAII for Working Directory:** `sh.change_dir()` ensures directory context is managed safely
- **`anyhow::Result` for Applications:** Error handling optimized for binaries, not libraries (thiserror would be wrong here)
- **Newtype for Project Root:** `project_root()` uses `PathBuf` with explicit parent traversal, avoiding string manipulation
- **Match Exhaustiveness:** The compiler guarantees all subcommands are handled - no runtime dispatch surprises

This replaces Makefiles/shell scripts with a testable, refactorable Rust binary. The `.cargo/config` alias integration makes `cargo xtask install` discoverable without PATH manipulation.

**Contribution Tip:**
When adding a new xtask subcommand:
1. Add variant to `flags::XtaskCmd` enum (xflags will generate the parser)
2. Implement `.run(&Shell)` method on the command struct
3. Add match arm in `main.rs`
4. The compiler enforces you handle all three - no forgotten branches

**Common Pitfalls:**
- **Anti-pattern:** Using `std::env::current_dir()` instead of Shell context - breaks composability
- **Anti-pattern:** Returning `()` instead of `Result` - loses error context for CI
- **Anti-pattern:** Hardcoding paths like `"./target"` - use `project_root().join()` for relocatability

**Related Patterns in Ecosystem:**
- **cargo-make:** More declarative (TOML) but less type-safe than xtask
- **just:** Command runner, but shell-based rather than Rust-native
- **xshell:** The `Shell` type used here - provides cross-platform scripting in Rust
- **Similar Usage:** `cargo-pgo`, `cargo-hakari`, and other workspace tools use this pattern

**Rust-Specific Insight:**
This pattern is ingenious in how it avoids proc-macro complexity:
- **Regular Macro (`xflags!`):** Generates code **in-place** rather than at compile-time, avoiding proc-macro crate separation
- **Two-Phase Workflow:**
  1. `xflags!` DSL defines structure (human-editable)
  2. `UPDATE_XFLAGS=1 cargo build` regenerates the Rust structs below
- **Self-Documenting:** The `// generated start` comment creates clear boundary between hand-written and generated code
- **Type Safety:** Generates exhaustive enums (`XtaskCmd`) that the compiler validates

Contrast with `clap` derive macros: xflags generates **visible** code you can step through in a debugger, while clap's proc macros are opaque. This aids debugging and learning.

**Contribution Tip:**
To add a new CLI flag:
1. Add the field in the `xflags!` DSL (e.g., `optional --new-flag value: String`)
2. Run `UPDATE_XFLAGS=1 cargo build -p xtask`
3. The generated struct will have the new field - compiler errors guide you to update call sites
4. Commit both the DSL and generated code (both are source-controlled)

**Common Pitfalls:**
- **Anti-pattern:** Editing the generated code directly - it will be overwritten next regen
- **Anti-pattern:** Forgetting to commit generated code after DSL changes - CI will fail
- **Anti-pattern:** Using environment variables for actual flags - `optional` in DSL is better than env vars

**Related Patterns in Ecosystem:**
- **clap derive:** More feature-rich but uses proc macros (compile-time black box)
- **structopt:** Predecessor to clap derive, similar trade-offs
- **pico-args:** Simpler but manual - no generation
- **xflags philosophy:** Inspired by Go's `flag` package - simple, visible, regenerative

```rust
impl flags::Codegen {
    pub(crate) fn run(self, _sh: &Shell) -> anyhow::Result<()> {
        match self.codegen_type.unwrap_or_default() {
            flags::CodegenType::All => {

            flags::CodegenType::AssistsDocTests => assists_doc_tests::generate(self.check),

    }
}
```

**Why This Matters for Contributors:** Commands implement a `.run(&Shell)` method rather than being free functions. This pattern provides dependency injection of the `Shell` context (from xshell crate) and enforces consistent error handling via `anyhow::Result`. The `self` consumption pattern ensures commands can't be accidentally re-run.

### Expert Rust Commentary

**Idiomatic Rating:** ⭐⭐⭐⭐⭐ (5/5 - Textbook Dependency Injection)

**Pattern Classification:** Trait-Based Command Pattern (Layer 3: Dependency Injection, Layer 6: Trait Patterns)

**Rust-Specific Insight:**
This demonstrates clean dependency injection without a DI framework:
- **`&Shell` as Injected Context:** All filesystem/process operations go through `Shell`, making commands testable with mock implementations
- **Consuming `self`:** The signature `fn run(self, sh: &Shell)` takes ownership, preventing accidental reuse:
  ```rust
  let cmd = Codegen { check: false, ... };
  cmd.run(&sh)?; // OK
  cmd.run(&sh)?; // Compiler error: value moved
  ```
- **`anyhow::Result` for Binaries:** Application-level error aggregation (vs `thiserror` for libraries)
- **`.unwrap_or_default()` Pattern:** Gracefully handles missing enum values without panicking

The pattern separates **what** to do (command struct) from **how** to do it (Shell context). This is testability nirvana.

**Contribution Tip:**
When adding a new command:
```rust
impl flags::NewCommand {
    pub(crate) fn run(self, sh: &Shell) -> anyhow::Result<()> {
        // Access sh.read_file(), sh.cmd!(), etc.
        // Return errors with .context() for better messages
        Ok(())
    }
}
```
The `Shell` reference prevents mocking hell - you can test with `MemoryShell` in unit tests.

**Common Pitfalls:**
- **Anti-pattern:** Using `&self` instead of `self` - allows accidental reuse with stale state
- **Anti-pattern:** Accessing filesystem directly via `std::fs` - breaks testability via Shell abstraction
- **Anti-pattern:** Returning `Result<(), Error>` with custom error type - use `anyhow::Result` in binaries

**Related Patterns in Ecosystem:**
- **clap's `Parser` trait:** Similar pattern - parse once, consume for execution
- **tower::Service:** More complex but same idea - `call(self, req)` consumes service
- **xshell::Shell:** The context type used here - provides cross-platform scripting API
- **anyhow vs thiserror:** This correctly uses anyhow for application code (vs thiserror for libraries)

## Pattern 4: AST Generation from Ungrammar with proc_macro2::quote
**File:** /Users/amuldotexe/Desktop/Pre20260126/OSS202601/Notes2026/rust-analyzer-patterns/xtask/src/codegen/grammar.rs
**Category:** Code Generation

**Code Example:**
```rust
fn generate_tokens(grammar: &AstSrc) -> String {
    let tokens = grammar.tokens.iter().map(|token| {
        let name = format_ident!("{}", token);
        let kind = format_ident!("{}", to_upper_snake_case(token));
        quote! {
            pub struct #name {
                pub(crate) syntax: SyntaxToken,
            }
            impl std::fmt::Display for #name {
                fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
                    std::fmt::Display::fmt(&self.syntax, f)
                }
            }
            impl AstToken for #name {
                fn can_cast(kind: SyntaxKind) -> bool { kind == #kind }
                fn cast(syntax: SyntaxToken) -> Option<Self> {
                    if Self::can_cast(syntax.kind()) { Some(Self { syntax }) } else { None }
                }
                fn syntax(&self) -> &SyntaxToken { &self.syntax }
            }
        }
    });

    add_preamble(
        crate::flags::CodegenType::Grammar,
        reformat(
            quote! {
                use std::{fmt, hash};
                use crate::{SyntaxKind::{self, *}, SyntaxToken, ast::AstToken};
                #(#tokens)*
            }
            .to_string(),
        ),
    )
}
```

**Why This Matters for Contributors:** This demonstrates using `proc_macro2` and `quote!` at compile-time (in an xtask, not a proc macro) to generate AST code from an ungrammar specification. The pattern chains: `quote! {}` → `.to_string()` → `reformat()` → `add_preamble()` → write to file. This is how rust-analyzer generates its entire syntax tree implementation from a declarative grammar.

**Rust-Specific Insight:**

- **Anti-pattern:** Not reformatting generated code - leads to style inconsistencies

## Pattern 5: Lowering Grammar Rules to AST Representation
**File:** /Users/amuldotexe/Desktop/Pre20260126/OSS202601/Notes2026/rust-analyzer-patterns/xtask/src/codegen/grammar.rs
**Category:** Code Generation

**Code Example:**
```rust
fn lower(grammar: &Grammar) -> AstSrc {
    let mut res = AstSrc {
        tokens:
            "Whitespace Comment String ByteString CString IntNumber FloatNumber Char Byte Ident"
                .split_ascii_whitespace()
                .map(|it| it.to_owned())
                .collect::<Vec<_>>(),
        ..Default::default()
    };

    let nodes = grammar.iter().collect::<Vec<_>>();

    for &node in &nodes {
        let name = grammar[node].name.clone();
        let rule = &grammar[node].rule;
        let _g = panic_context::enter(name.clone());
        match lower_enum(grammar, rule) {
            Some(variants) => {
                let enum_src = AstEnumSrc { doc: Vec::new(), name, traits: Vec::new(), variants };
                res.enums.push(enum_src);
            }
            None => {
                let mut fields = Vec::new();
                lower_rule(&mut fields, grammar, None, rule);
                res.nodes.push(AstNodeSrc { doc: Vec::new(), name, traits: Vec::new(), fields });
            }
        }
    }

    deduplicate_fields(&mut res);
    extract_enums(&mut res);
    extract_struct_traits(&mut res);
    extract_enum_traits(&mut res);
    res
}
```

**Why This Matters for Contributors:** The "lowering" pattern transforms a high-level grammar (ungrammar) into a structured IR (AstSrc) before code generation. This separation of concerns allows complex transformations like trait extraction and field deduplication. The `panic_context::enter(name)` provides better error messages during code generation.

### Expert Rust Commentary

**Idiomatic Rating:** ⭐⭐⭐⭐⭐ (5/5 - Compiler-Style IR Transformation)

**Pattern Classification:** Multi-Phase Lowering (Layer 7: Complex Domain Model Support)

**Rust-Specific Insight:**
This mirrors how rustc itself works - progressive lowering through multiple IRs:
1. **Source:** Ungrammar DSL (declarative, human-written)
2. **IR:** `AstSrc` (structured, ready for transformations)
3. **Output:** Generated Rust code (executable)

Key idioms:
- **Builder Pattern with `..Default::default()`:** Partially populate struct, let Default fill the rest
- **Multi-Pass Transformations:** Each function (`deduplicate_fields`, `extract_enums`, etc.) makes one focused change
- **Panic Context:** `let _g = panic_context::enter(name)` adds context to panics (similar to `#[track_caller]`)
- **Separation of Concerns:** Lowering logic is isolated from code generation (`quote!`)

The IR transformation pipeline:
```
Grammar → lower() → AstSrc → deduplicate_fields() → extract_enums() →
extract_struct_traits() → extract_enum_traits() → AstSrc (final)
```

To add a new transformation pass:

```rust
}

}
```

**Common Pitfalls:**
- **Anti-pattern:** Transforming grammar directly to code - loses ability to do cross-node optimizations
- **Anti-pattern:** Using `panic!()` without context - use `panic_context::enter()` for better error messages
- **Anti-pattern:** Ordering transformations incorrectly - e.g., extracting traits before deduplicating fields

**Related Patterns in Ecosystem:**
- **rustc's HIR/MIR/LLVM IR:** Same multi-phase lowering approach
- **salsa queries:** rust-analyzer uses salsa for incremental computation, this is the batch version

## Pattern 6: Trait Extraction from Common Fields
**File:** /Users/amuldotexe/Desktop/Pre20260126/OSS202601/Notes2026/rust-analyzer-patterns/xtask/src/codegen/grammar.rs
**Category:** Code Generation

**Code Example:**
```rust
const TRAITS: &[(&str, &[&str])] = &[
    ("HasAttrs", &["attrs"]),
    ("HasName", &["name"]),
    ("HasVisibility", &["visibility"]),
    ("HasGenericParams", &["generic_param_list", "where_clause"]),
    ("HasGenericArgs", &["generic_arg_list"]),
    ("HasTypeBounds", &["type_bound_list", "colon_token"]),
    ("HasModuleItem", &["items"]),
    ("HasLoopBody", &["label", "loop_body"]),
    ("HasArgList", &["arg_list"]),
];

fn extract_struct_traits(ast: &mut AstSrc) {
    for node in &mut ast.nodes {
        for (name, methods) in TRAITS {
            extract_struct_trait(node, name, methods);
        }
    }
}

fn extract_struct_trait(node: &mut AstNodeSrc, trait_name: &str, methods: &[&str]) {
    let mut to_remove = Vec::new();
    for (i, field) in node.fields.iter().enumerate() {
        let method_name = field.method_name();
        if methods.iter().any(|&it| it == method_name) {
            to_remove.push(i);
        }
    }
    if to_remove.len() == methods.len() {
        node.traits.push(trait_name.to_owned());
        node.remove_field(to_remove);
    }
}
```

**Why This Matters for Contributors:** This pattern analyzes generated AST nodes to discover common field patterns and factor them into traits. It's a form of automatic DRY enforcement: instead of manually implementing `HasName` on 50 nodes, the code generator detects which nodes have a `name` field and generates trait impls. This reduces generated code size and ensures consistency.

### Expert Rust Commentary

**Idiomatic Rating:** ⭐⭐⭐⭐⭐ (5/5 - Metaprogramming Excellence)

**Pattern Classification:** Automatic Trait Derivation via Analysis (Layer 6: Trait Patterns, Layer 17: Code Generation)

**Rust-Specific Insight:**
This is **compile-time reflection** - analyzing code structure to generate trait impls:
- **Data-Driven Traits:** The `TRAITS` table maps trait names to required fields
- **Automatic Factoring:** If a node has all required fields (`attrs`), it gets the trait (`HasAttrs`)
- **Removal After Extraction:** Fields are removed from node definition to avoid duplication (DRY)
- **All-or-Nothing:** `to_remove.len() == methods.len()` ensures partial matches don't create broken traits

Without this pattern, you'd manually write:
```rust
impl HasName for FnDef { ... }
impl HasName for StructDef { ... }
// ... 50 more impls
```

With this pattern: compiler does it automatically if node has `name` field.

**Contribution Tip:**
To add a new factored trait:
1. Add to `TRAITS` table:
   ```rust
   ("HasDocComments", &["doc_comments"]),
   ```
2. Run `cargo xtask codegen grammar`
3. All nodes with `doc_comments` field will implement `HasDocComments`
4. The field is removed from node structs (moved to trait)

To check which nodes get a trait: search generated code for `impl HasName for`.

**Common Pitfalls:**
- **Anti-pattern:** Partial field matches - if node has only 1 of 2 required fields, trait isn't implemented (good!)
- **Anti-pattern:** Forgetting to remove fields after trait extraction - leads to duplicate accessors
- **Anti-pattern:** Using this for traits with behavior - only works for field accessors
- **Anti-pattern:** Not running codegen after grammar changes - traits get out of sync

**Related Patterns in Ecosystem:**
- **Rust derive macros:** Similar automatic impl generation, but based on `#[derive]` attributes
- **serde's `#[serde(flatten)]`:** Flattens fields into parent struct, opposite direction
- **AST rewriting:** Tools like `syn` can analyze Rust code similarly
- **Similar:** `salsa`'s query groups automatically generate trait impls

## Pattern 7: Reformat and Preamble Utilities for Generated Code
**File:** /Users/amuldotexe/Desktop/Pre20260126/OSS202601/Notes2026/rust-analyzer-patterns/xtask/src/codegen.rs
**Category:** Code Generation

**Code Example:**
```rust
fn reformat(text: String) -> String {
    let sh = Shell::new().unwrap();
    let rustfmt_toml = project_root().join("rustfmt.toml");
    let version = cmd!(sh, "rustup run stable rustfmt --version").read().unwrap_or_default();

    let mut stdout = if !version.contains("stable") {
        cmd!(sh, "rustfmt --config-path {rustfmt_toml} --config fn_single_line=true")
            .stdin(text)
            .read()
            .unwrap()
    } else {
        cmd!(
            sh,
            "rustup run stable rustfmt --config-path {rustfmt_toml} --config fn_single_line=true"
        )
        .stdin(text)
        .read()
        .unwrap()
    };
    if !stdout.ends_with('\n') {
        stdout.push('\n');
    }
    stdout
}

fn add_preamble(cg: CodegenType, mut text: String) -> String {
    let preamble = format!("//! Generated by `cargo xtask codegen {cg}`, do not edit by hand.\n\n");
    text.insert_str(0, &preamble);
    text
}
```

**Why This Matters for Contributors:** Generated code should be formatted and have clear provenance. The `reformat()` function pipes generated code through `rustfmt` with project-specific config. The `add_preamble()` function adds a comment explaining how to regenerate the file. This pattern makes generated code readable and maintainable.

```rust
fn ensure_file_contents(cg: CodegenType, file: &Path, contents: &str, check: bool) -> bool {
    let contents = normalize_newlines(contents);
    if let Ok(old_contents) = fs::read_to_string(file)
        && normalize_newlines(&old_contents) == contents
    {
        // File is already up to date.
        return false;
    }

    let display_path = file.strip_prefix(project_root()).unwrap_or(file);
    if check {
        panic!(
```

## Pattern 9: Comment Block Extraction for Documentation Generation
**File:** /Users/amuldotexe/Desktop/Pre20260126/OSS202601/Notes2026/rust-analyzer-patterns/xtask/src/codegen.rs
**Category:** Code Generation

**Code Example:**
```rust
#[derive(Clone)]
pub(crate) struct CommentBlock {
    pub(crate) id: String,
    pub(crate) line: usize,
    pub(crate) contents: Vec<String>,
    is_doc: bool,
}

impl CommentBlock {
    fn extract(tag: &str, text: &str) -> Vec<CommentBlock> {
        assert!(tag.starts_with(char::is_uppercase));

        let tag = format!("{tag}:");
        let mut blocks = CommentBlock::extract_untagged(text);
        blocks.retain_mut(|block| {
            let first = block.contents.remove(0);
            let Some(id) = first.strip_prefix(&tag) else {
                return false;
            };

            if block.is_doc {
                panic!("Use plain (non-doc) comments with tags like {tag}:\n    {first}");
            }

            id.trim().clone_into(&mut block.id);
            true
        });
        blocks
    }

    fn extract_untagged(text: &str) -> Vec<CommentBlock> {
        let mut res = Vec::new();
        let lines = text.lines().map(str::trim_start);
        let dummy_block =
            CommentBlock { id: String::new(), line: 0, contents: Vec::new(), is_doc: false };
        let mut block = dummy_block.clone();
        for (line_num, line) in lines.enumerate() {
            match line.strip_prefix("//") {
                Some(mut contents) if !contents.starts_with('/') => {
                    if let Some('/' | '!') = contents.chars().next() {
                        contents = &contents[1..];
                        block.is_doc = true;
                    }
                    if let Some(' ') = contents.chars().next() {
                        contents = &contents[1..];
                    }
                    block.contents.push(contents.to_owned());
                }
                _ => {
                    if !block.contents.is_empty() {
                        let block = mem::replace(&mut block, dummy_block.clone());
                        res.push(block);
                    }
                    block.line = line_num + 2;
                }
            }
        }
        if !block.contents.is_empty() {
            res.push(block);
        }
        res
    }
}
```

**Why This Matters for Contributors:** This pattern extracts structured metadata from source code comments using a tagged format (e.g., `// Assist: rename_variable`). It powers documentation generation from inline examples in assist handlers. The pattern distinguishes doc comments (`///`, `//!`) from tagged metadata comments (`//`) to avoid confusion.

```rust
fn get_lints_as_text(stdout: &str) -> (
    impl Iterator<Item = (String, &str, Severity)> + '_,
    impl Iterator<Item = (String, Lint, impl Iterator<Item = String> + '_)> + '_,
) {
    let lints_pat = "----  -------  -------\n";
    let lint_groups_pat = "----  ---------\n";
    let lints = find_and_slice(stdout, lints_pat);
    let lint_groups = find_and_slice(lints, lint_groups_pat);

    let lints = lints.lines().take_while(|l| !l.is_empty()).map(|line| {
        let (name, rest) = line.trim().split_once(char::is_whitespace).unwrap();
        let (severity, description) = rest.trim().split_once(char::is_whitespace).unwrap();
        (name.trim().replace('-', "_"), description.trim(), severity.parse().unwrap())
    });

    let lint_groups = lint_groups.lines().take_while(|l| !l.is_empty()).map(|line| {
        let (name, lints) = line.trim().split_once(char::is_whitespace).unwrap();
        let label = name.trim().replace('-', "_");
        let lint = Lint {
            description: format!("lint group for: {}", lints.trim()),
            default_severity: Severity::Allow,
            warn_since: None,
            deny_since: None,
        };
        let children = lints
            .split_ascii_whitespace()
            .map(|s| s.trim().trim_matches(',').replace('-', "_"));
        (label, lint, children)
    });

    (lints, lint_groups)
}
```

```rust
fn dist_server(
    sh: &Shell,
    release: &str,
    target: &Target,
    allocator: Malloc,

    dev_rel: bool,

}
```

## Pattern 14: Profile-Guided Optimization (PGO) Workflow

```rust
pub(crate) fn gather_pgo_profile<'a>(
    sh: &'a Shell,
    ra_build_cmd: Cmd<'a>,
    target: &str,
    train_crate: PgoTrainingCrate,
) -> anyhow::Result<PathBuf> {
    let pgo_dir = std::path::absolute("rust-analyzer-pgo")?;
    if pgo_dir.is_dir() {
        std::fs::remove_dir_all(&pgo_dir)?;
    }
    std::fs::create_dir_all(&pgo_dir)?;

    // Build RA with PGO instrumentation
    let cmd_gather =
        ra_build_cmd.env("RUSTFLAGS", format!("-Cprofile-generate={}", pgo_dir.to_str().unwrap()));
    cmd_gather.run().context("cannot build rust-analyzer with PGO instrumentation")?;

    let (train_path, label) = match &train_crate {
        PgoTrainingCrate::RustAnalyzer => (PathBuf::from("."), "itself"),
        PgoTrainingCrate::GitHub(repo) => {
            (download_crate_for_training(sh, &pgo_dir, repo)?, repo.as_str())
        }
    };

    // Run RA on training data to generate profiles
    eprintln!("Training RA on {label}...");
    cmd!(
        sh,
        "target/{target}/release/rust-analyzer analysis-stats -q --run-all-ide-things {train_path}"
    )
    .run()
    .context("cannot generate PGO profiles")?;

    // Merge profiles into a single file
    let merged_profile = pgo_dir.join("merged.profdata");
    let profile_files = std::fs::read_dir(pgo_dir)?
        .filter_map(|entry| {
            let entry = entry.ok()?;
            if entry.path().extension() == Some(OsStr::new("profraw")) {
                Some(entry.path().to_str().unwrap().to_owned())
            } else {
                None
            }
        })
        .collect::<Vec<_>>();

    cmd!(sh, "{llvm_profdata} merge {profile_files...} -o {merged_profile}").run()?;

    Ok(merged_profile)
}
```

**Why This Matters for Contributors:** PGO can provide 10-30% performance improvements for compiler-like tools. This pattern automates the three-step process: (1) build with instrumentation, (2) run on representative workload, (3) merge profiles. The ability to train on external crates (`PgoTrainingCrate::GitHub`) allows profiling against real-world code.

```rust
    path: PathBuf,
    original_contents: String,

impl Patch {
    fn new(sh: &Shell, path: impl Into<PathBuf>) -> anyhow::Result<Patch> {
        let path = path.into();
        let contents = sh.read_file(&path)?;
        Ok(Patch { path, original_contents: contents.clone(), contents })
    }

    fn replace(&mut self, from: &str, to: &str) -> &mut Patch {
        assert!(self.contents.contains(from));
        self.contents = self.contents.replace(from, to);
        self
    }

    fn commit(&self, sh: &Shell) -> anyhow::Result<()> {
        sh.write_file(&self.path, &self.contents)?;
        Ok(())
    }
}

        .replace(r#""releaseTag": null"#, &format!(r#""releaseTag": "{release_tag}""#))
        .replace(r#""enabledApiProposals": [],"#, r#""#);
    patch.commit(sh)?;

    Ok(())
}
```

```rust
fn check_licenses(sh: &Shell) {
    const EXPECTED: &[&str] = &[
        "(MIT OR Apache-2.0) AND Unicode-3.0",

        "Apache-2.0 WITH LLVM-exception",

    if licenses != expected {

            if !expected.contains(&l) {
                diff += &format!("  {l}\n")
            }
        }
        diff.push_str("\nMissing Licenses:\n");
        for l in expected {
            if !licenses.contains(&l) {
                diff += &format!("  {l}\n")
            }
        }
        panic!("different set of licenses!\n{diff}");
    }
}
```
