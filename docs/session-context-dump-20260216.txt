SECTION 3: TREE-SITTER CAPABILITIES ANALYSIS
================================================================================

What tree-sitter extracts that we currently throw away:

FOR RUST:
  - type_parameters → generic bounds are trait dependency edges
  - where_clause → more trait bounds
  - visibility_modifier → pub/pub(crate)/pub(super)
  - function_modifiers → async, unsafe, const
  - attribute_item → #[derive(...)], #[cfg(...)], #[test], #[async_trait]
  - return_type → type information
  - parameter with type → function signature
  - use_declaration → full import paths
  - closure_expression → closure params, body, move keyword
  - lifetime → 'a, 'static
  - macro_invocation → which macros used
  - impl_item → trait field + type field
  - match_expression with match_arm → pattern info
  - unsafe_block → unsafe code locations

FOR TYPESCRIPT:
  - type annotations, interfaces, generics

FOR GO:
  - interface definitions, goroutine spawns

FOR JAVA:
  - annotations, generics, interface implementations

FOR C/C++:
  - preprocessor directives, templates, header includes

Tree-sitter query language example:
  (function_item
    (visibility_modifier) @vis
    (function_modifiers) @mods
    name: (identifier) @name
    type_parameters: (type_parameters
      (constrained_type_parameter
        name: (identifier) @generic
        bounds: (trait_bounds) @bounds))
    parameters: (parameters
      (parameter
        pattern: (identifier) @param_name
        type: (_) @param_type))
    return_type: (_) @ret_type)

Tree-sitter features not used:
  - Incremental parsing (re-parse only changed bytes)
  - Language injection (SQL in Rust strings, JS in HTML)
  - tree-sitter-tags crate (purpose-built for def/ref extraction)
  - Predicates in queries (#match?, #eq?, #is?)

SECTION 4: ASCENT DATALOG ENGINE
================================================================================

Ascent crate v0.8.0 — proc-macro Datalog for Rust.
Compiles rules into efficient Rust code at compile time.
Relations are Vec<(T1, T2, ...)> under the hood.

Example Ascent program for code analysis:

  ascent! {
      struct TraitHierarchyProgram;
      relation supertrait(String, String);
      relation implements(String, String);
      relation all_supertraits(String, String);
      relation all_implementors(String, String);

      all_supertraits(t, s) <-- supertrait(t, s);
      all_supertraits(t, s2) <-- all_supertraits(t, s1), supertrait(s1, s2);
      all_implementors(tr, ty) <-- implements(ty, tr);
      all_implementors(tr, ty) <-- all_supertraits(sub, tr), implements(ty, sub);
  }

Why Datalog beats alternatives for code analysis:
  1. Recursive queries first-class, guaranteed termination
  2. Rules compose naturally (layer on layer)
  3. Incremental maintenance possible (delta propagation)
  4. 2-3 lines vs 12-15 lines for equivalent SQL/Rust

SECTION 5: CROSS-LANGUAGE BOUNDARY PATTERNS
================================================================================

Pattern 1: FFI (Rust <-> C/C++)
  Rust: extern "C" { fn rocksdb_put(...); }
  C: int rocksdb_put(...) { ... }
  Detection: function name + ABI marker

Pattern 2: WASM (Rust -> JS/TS)
  Rust: #[wasm_bindgen] pub fn process_data(...)
  JS: import { process_data } from './pkg/my_module'
  Detection: attribute + import name + .d.ts files from wasm-bindgen

Pattern 3: PyO3/JNI (Rust -> Python/Java)
  Rust: #[pyfunction] fn analyze_code(...)
  Python: from parseltongue import analyze_code
  Detection: attribute + import from module

Pattern 4: gRPC/HTTP (Rust <-> Any)
  Rust: #[post("/api/v1/analyze")] async fn analyze(...)
  Client: requests.post("/api/v1/analyze", ...)
  Detection: string literal URL paths in both sides

Pattern 5: Message Queues (Rust <-> Any via Iggy/Kafka)
  Rust: client.send_messages("user-events", ...)
  Java: consumer.subscribe("user-events")
  Detection: string literal topic names in function arguments

Key insight: Tree-sitter sees BOTH sides. Rust-analyzer sees only Rust.
Cross-language edges need tree-sitter for detection, Ascent for joining.

SECTION 6: STORAGE ARCHITECTURE RESEARCH (from prior session)
================================================================================

15+ storage options evaluated across 6 categories:

Option A: Force everything into CozoDB
  Pro: No new dependencies
  Con: Untyped DataValue blobs, can't query type-level data

Option B: Two CozoDBs (syntactic + semantic)
  Pro: Separates concerns
  Con: Two databases to maintain, still untyped

Option C: Query Salsa/rust-analyzer directly
  Pro: No storage layer
  Con: Requires rust-analyzer running, not persistable

Option D: Hybrid (CozoDB + Custom Rust TypedStore + Ascent)
  Pro: Pragmatic, keeps existing code
  Con: Two storage systems to coordinate

Option E: DuckDB replaces CozoDB entirely
  Pro: SQL + recursive CTEs + DuckPGQ graph + Parquet export
  Con: +50MB binary via FFI, not pure Rust

Recommendation from research: Option D for v1.7.3, but user pivoted to v2.0.0
which is essentially "Option D minus CozoDB" = pure Ascent + TypedStore.

CozoDB dormancy risk: last release mid-2023, unanswered issues accumulating.
Kuzu eliminated: Apple acquisition Oct 2025, repo archived.

SECTION 7: ARCHITECTURAL DECISIONS FOR v2.0.0
================================================================================

Decision 1: NO BACKWARD COMPATIBILITY
  v2.0.0 is a clean break. New JSON shapes, new endpoints, new CLI.

Decision 2: NO OLD CODE DELETED
  v1.x stays in repo, compiles, works. v2.0.0 is additive.

Decision 3: CozoDB REMOVED
  No CozoDB dependency in v2.0.0. Pure Rust data structures.
  HashMap + secondary indexes + MessagePack serialization.

Decision 4: NEW CRATE PREFIX
  All new crates use rust-llm-* prefix. pt* prefix frozen.

Decision 5: INDEPENDENT CRATES
  v2.0.0 crates have ZERO dependency on v1.x crates.
  They coexist in the same workspace but share nothing.

Decision 6: TREE-SITTER QUERIES
  Replace cursor walking with declarative tree-sitter queries.
  Extract 10x more data from the same parse trees.

Decision 7: ASCENT FOR REASONING
  All recursive analysis via Ascent Datalog rules.
  Graph algorithms rewritten on typed data.

Decision 8: RUST-ANALYZER AS LIBRARY
  rust-analyzer (ra_ide, ra_hir) for Rust-only semantic depth.
  Tree-sitter for all 12 languages syntactic width.
