  Key Research Findings (Your Point is Validated)

  You're right - STDIO is the way to go for Parseltongue:

  | Transport | MCP Support | Use Case                                    |
  |-----------|-------------|---------------------------------------------|
  | STDIO     | 10/10 IDEs  | Local CLI, single-user, microsecond latency |
  | HTTP      | 9/10 IDEs   | Cloud/enterprise (future option)            |
  | SSE       | Deprecated  | Legacy only                                 |

  STDIO Advantages for Parseltongue:
  1. Universal compatibility - Works with ALL agentic IDEs (Cursor, VS Code,
   Windsurf, etc.)
  2. Zero network overhead - Microsecond response times
  3. Security - No exposed ports, no auth complexity
  4. Simplicity - Rust binary → STDIO → JSON-RPC 2.0

  Critical Constraint (Cursor):
  - 40 tool hard limit - Keep Parseltongue tools under 40 total
  - This affects ~10%+ of market

  Recommended MCP Command:
  parseltongue mcp --workspace ./my-project

  The MCP server manages base.db (git HEAD) and live.db (current state)
  internally, exactly as we discussed for state management.

  Architecture Overview

  ┌─────────────────────────────────────────────────────────────┐

  └──────────────────────────┬──────────────────────────────────┘
                             │

  └──────────────────────────┬──────────────────────────────────┘
                             │
                             ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  Backend API (HTTP - different from MCP!)                   │
  │  POST /analyze { repo: "facebook/react" }                   │
  │  GET  /status/:job_id                                       │
  │  GET  /graph/:repo_hash                                     │
  └──────────────────────────┬──────────────────────────────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐
  │ Job Queue     │  │ Cache Layer   │  │ Storage       │
  │ (Redis/SQS)   │  │ (Redis)       │  │ (S3/RocksDB)  │
  └───────────────┘  └───────────────┘  └───────────────┘
          │
          ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  Worker Pool                                                │
  │  1. git clone --depth 1 (shallow clone)                     │
  │  2. parseltongue index --path /tmp/repo --output graph.db   │
  │  3. Upload results to storage                               │
  │  4. Cleanup temp files                                      │
  └─────────────────────────────────────────────────────────────┘

  Key Components Needed

  1. Git Cloner Service

  User input: github.com/facebook/react
     │
     ▼
  Shallow clone: git clone --depth 1 --branch main
     │
     ▼
  Temp directory: /tmp/jobs/{job_id}/repo

  2. Job Queue (for async processing)

  - Large repos take time (React = 2000+ files)
  - Queue jobs, return job_id immediately
  - WebSocket/SSE for progress updates

  3. Cache Strategy

  Cache key: SHA256(repo_url + default_branch + last_commit_sha)
  TTL: 24 hours (or webhook-based invalidation)

  Hit?  → Return cached graph instantly
  Miss? → Queue analysis job

  4. Cost Considerations

  | Component  | Challenge                         | Solution
     |
  |------------|-----------------------------------|------------------------
  ---|
  | Clone time | Large repos slow                  | Shallow clone (--depth 
  1) |
  | Disk space | Repos accumulate                  | Temp files + cleanup
     |
  | CPU        | Parsing is intensive              | Worker pool + queue
     |
  | Storage    | Graph DBs grow                    | Compressed JSON + TTL
     |
  | Abuse      | Someone indexes linux kernel 100x | Rate limiting + cache
     |

  5. Rate Limiting

  - Per-IP: 10 requests/hour
  - Per-repo: Cache for 24h
  - Repo size limit: < 500MB, < 10K files

  Two Deployment Options

  Option A: Serverless (Lower cost, slower)

  Vercel/Cloudflare → Lambda/Workers → S3
  - Cold start penalty
  - 15 min timeout risk for large repos
  - ~$0 for light usage

  Option B: Always-on Workers (Faster, fixed cost)

  Fly.io / Railway → Redis Queue → Workers
  - No cold start
  - Handles large repos
  - ~$20-50/month

  Minimum Viable Version

  1. Frontend: Single page with URL input + graph viewer
  2. Backend:  Single server with parseltongue binary
  3. Storage:  Local disk with LRU eviction
  4. Cache:    In-memory HashMap (repo_url → graph_json)

  This is NOT the MCP use case - this is HTTP API for web service,
  completely separate from the local STDIO MCP server we discussed for
  Claude Code integration.

  WASM-First Architecture

  ┌─────────────────────────────────────────────────────────────┐
  │  User's Browser                                              │
  │                                                              │
  │  ┌────────────────────────────────────────────────────────┐ │
  │  │  1. User pastes: github.com/facebook/react             │ │
  │  │  2. JS fetches repo via GitHub API / zipball           │ │
  │  │  3. Parseltongue WASM parses in-browser                │ │
  │  │  4. Graph rendered with react-force-graph-3d           │ │
  │  └────────────────────────────────────────────────────────┘ │
  │                                                              │
  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
  │  │ parseltongue │  │ IndexedDB    │  │ WebGL Graph      │  │
  │  │ .wasm        │  │ (cache)      │  │ Renderer         │  │
  │  │ (~2-5MB)     │  │              │  │                  │  │
  │  └──────────────┘  └──────────────┘  └──────────────────┘  │
  └─────────────────────────────────────────────────────────────┘

  Server: Just a CDN serving static files!

  Why This Is Better

  | Aspect       | Server-Side              | WASM Client-Side     |
  |--------------|--------------------------|----------------------|
  | Hosting cost | $20-50/month             | ~$0 (static CDN)     |
  | Scaling      | You pay for compute      | User's CPU pays      |
  | Privacy      | Code goes to your server | Code stays local     |
  | Latency      | Network round-trip       | Instant (after load) |
  | Complexity   | Queue, workers, storage  | Just WASM + JS       |
